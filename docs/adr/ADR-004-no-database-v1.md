# ADR-004: No Database in v1 — File-Based Architecture

- **Status**: Accepted
- **Related RFCs**: — *(stream A — decided in concept phase, see OC-04)*
- **Related PRDs**: `docs/prd/PRD-001-data-ingestion-pipeline.md`

## Context & Problem Statement

A daily pipeline producing PNG renders and JSON data files needs a way to track what exists, serve it to the browser, and index it for the gallery. The question is whether this requires a database or whether file-based structures are sufficient at v1 scale.

## Decision

No database in v1. Everything is file-shaped. State, configuration, and indexes all live in JSON/YAML files. `manifest.json` is the gallery index, rebuilt from scratch after each pipeline run. SQLite deferred to if `manifest.json` grows unmanageable.

## Rationale

The data produced by the pipeline is naturally file-shaped. The gallery index is a flat list of renders with metadata — a JSON file, not a relational query. Recipes are YAML files — named, versioned, committed to git. Adding a database introduces a stateful service that must be backed up, migrated, and kept in sync with the filesystem. At v1 scale this complexity is not justified.

## Alternatives Considered

1. **SQLite for manifest and state**
   - **Pros**: Queryable, transactional, handles concurrent access
   - **Cons**: Stateful artifact requiring separate backup. Migration required when schema changes. Overkill for a flat list of renders.
   - **Why Rejected**: Not needed at v1 scale. Deferred if `manifest.json` grows beyond manageable size.

2. **PostgreSQL**
   - **Why Rejected**: Significant operational overhead for data that is naturally file-shaped. Prefect Server already runs Postgres internally — a second instance for application data is not justified.

## Consequences

**Positive:**
- Zero database operational overhead in v1
- Everything in the filesystem is inspectable, diffable, and portable
- `manifest.json` regenerated by walking `renders/` — no database restore needed
- Self-hosters need only `docker compose up` — no database seeding step

**Negative:**
- `manifest.json` rebuilt entirely on each pipeline run — at very large scale this becomes slow
- No query capabilities across render metadata — filtering done in the React gallery

**Neutral:**
- Prefect Server runs its own internal Postgres. Scoped to Prefect, not the application database.

## Implementation Notes

`manifest.json` at `renders/manifest.json`. Rebuilt by Task 06 (index). Format: array of render objects with recipe name, date, sources, render type, file path, and key statistics.
